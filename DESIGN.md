# Design conventions

Reference for the patterns already established in `content-hub-firebase.html`. Follow these when adding new fields, panels, or icons so the app keeps reading as one system rather than accreting one-off styles. This is documentation only — it doesn't run; edit the source file and regenerate `index.html` as usual.

## Colors

All color is driven by CSS custom properties on `:root`, with dark as the default and explicit overrides for `:root[data-theme="dark"]` / `:root[data-theme="light"]` (set by the manual theme toggle, persisted to localStorage). Never hardcode a hex color in a new rule — use the existing tokens:

- `--brand`, `--brand-soft`, `--brand-gradient-h` — MediaShock orange, used for active states, primary buttons, links, focus rings.
- `--surface`, `--surface-alt` — card/modal background vs. sunken field background.
- `--border` — all hairlines and input borders.
- `--text`, `--text-soft`, `--text-faint` — primary copy, field labels, secondary/meta text (in that descending order of emphasis).
- `--danger`, `--danger-soft`, `--good`, `--bad` — status colors (delete actions, goal pacing, etc.)

## Spacing rhythm

Three tiers, used consistently everywhere a label sits above a field or a group of fields sits above another group:

- **Label → field: `0.35rem`** (`.field-col`, and the gap baked into `#postForm label`/`#ideaForm label`). Never let a label sit as a bare sibling of its field — always wrap it in `.field-col` so it picks up this tight gap instead of a section gap.
- **Within a section: `0.9rem`** (`.form-section`'s own gap) — spacing between fields that belong to the same logical group.
- **Between sections: divider + `1.3rem`** (`.form-section + .form-section`, `.form-section + .modal-actions`) — a hairline `border-top` plus `1.1rem` padding-top, so unrelated groups of fields read as visually distinct blocks instead of one long stack.

## Form sections

The Post and Idea modals are each broken into `.form-section` blocks (currently: Details / Owners / Content for Post; Details / Brief / Stage & Feedback for Idea — the Tasks sections were removed, see below). Sections are separated by whitespace and a hairline divider only (`.form-section + .form-section`) — **no heading text on the section itself.** An earlier version added an uppercase eyebrow label per section, but that was redundant with the fields' own labels and read as clutter; it was removed. When adding a new field, put it in the section it semantically belongs to (or give it its own `.form-section` if none fit), but every field/group still needs its *own* `<label>` — don't rely on the section boundary alone to convey what a block is for. (Fixed one regression from the original section-label removal: Idea's Tasks block had no field label of its own, only the section heading — it now has `<label>Tasks</label>` matching Post's.)

## Icons

All icons in this app -- not just icon-only buttons, any status/type indicator too (activity feed, image-count badges, goal pace, link chips) -- are inline SVG, never emoji or an icon font. This was enforced retroactively in a full sweep; if you ever find a stray emoji being used as an icon, replace it the same way. Convention: `stroke="currentColor"`, `fill="none"`, `stroke-width="1.8"`–`2.4` (bump toward the higher end only for small badge-style icons that need to read at a glance), `viewBox="0 0 24 24"`, rendered at `16x16` for `.icon-btn` buttons or `12x12`–`14x14` for inline status icons. Wrap clickable icon-only buttons in `.icon-btn` (2.1rem square, 1px border, 8px radius, `--surface` background), or a variant like `.emoji-inline-btn` when it needs to sit inside/overlay another field.

**Shared icon constants**: reusable icon markup lives as `var ICON_*` string constants (`ICON_X`, `ICON_X_SMALL`, `ICON_CHECK`, `ICON_CHECK_CIRCLE`, `ICON_ALERT_TRIANGLE`, `ICON_CALENDAR`, `ICON_EDIT`, `ICON_TRASH`, `ICON_LIGHTBULB`, `ICON_ARROW_RIGHT`, `ICON_MESSAGE`, `ICON_LINK`, `ICON_IMAGE`, `ACTIVITY_ICON_DOT`), defined together in one place right before `ACTIVITY_ICONS` (search "shared inline icons"). Reuse one of these for any new icon before drawing a new SVG from scratch. **Ordering gotcha**: these must stay declared *before* anything that builds an object/lookup out of them at module top-level (like `ACTIVITY_ICONS`) -- `var` hoisting only hoists the declaration, not the assignment, so referencing a not-yet-assigned `ICON_*` constant at module-init time silently evaluates to `undefined` (this exact bug shipped once: `ACTIVITY_ICONS` was built above the icon constants, so every activity type silently fell back to the dot icon). Using an `ICON_*` constant *inside* a function body is always safe regardless of where in the file that function is defined, since the function doesn't run until later, after all top-level vars are assigned.

**Deliberate exceptions** (real emoji, left as-is on purpose -- don't "fix" these): the emoji picker's own category data (`EMOJI_CATEGORIES`) is the feature's content, not UI chrome. The LinkedIn post preview's action row (👍 Like / 💬 Comment / 🔁 Repost / ✈️ Send) is simulating LinkedIn's actual UI as accurately as possible, not our own app's iconography -- converting it to our monotone style would make the preview less accurate, not more consistent.

## Reusable input patterns

- **Chips-with-add-row**: owners use `createOwnerChipInput(inputEl, chipsEl, onChange)` — a small factory returning `{add, set, get}`. Reuse this factory (don't hand-roll a new chip list) for anything that's "add a short label, show it as a removable pill."
- **Checklist-with-assignee**: tasks use `createTaskListInput(listEl, inputEl, assigneeSelectEl, getOwners)` similarly — checkbox + text + avatar + remove, assignee dropdown synced live to the item's current owners.
- **Emoji picker**: `createEmojiPicker(toggleBtn, pickerEl, tabsEl, gridEl, textarea, onInsert)` — instantiate one per textarea that should support inserting an emoji (Post Copy, Idea/Brief). The trigger button lives inside a `.textarea-emoji-wrap` (position: relative wrapper around the textarea), overlaid bottom-right via `.emoji-inline-btn`, so it reads as part of the text field rather than a separate toolbar action.
- **Avatars**: `avatarHtml(name, sizeClass)` — always derive color and initials from `avatarColorFor(name)` / `getInitials(name)`, both hashing the trimmed/lowercased name, so the same person renders identically everywhere in the app. Never assign an avatar color ad hoc.
- **Image links**: both Post and Idea images are entered as pasted URLs (`.image-link-row` input + Add button, `.image-link-grid` of `.image-thumb` tiles with a `.image-remove` button), not uploaded files — deliberately. Firestore documents have a hard 1MiB limit, and embedding uploaded images as base64 forced heavy compression that was visibly pixelated; a URL is a few bytes regardless of the source image's real quality. A thumb gets `.broken` (via the `<img>`'s `error` event) when its URL doesn't load — note `.image-thumb.broken::after`'s decorative placeholder icon must keep `pointer-events: none`, or it silently eats clicks meant for the remove button underneath it. Thumbs are `draggable="true"` (with `draggable="false"` on the inner `<img>`/`<a>` so the browser doesn't hijack the drag into an image-drag instead) — `dragstart`/`dragover`/`drop` on `.image-thumb` reorder the images array via `.splice()`. Post and Idea now share the exact same classes/behavior (`renderPostImageGrid()` / `renderIdeaImageGrid()` are separate functions with separate state, `currentImages` vs `currentIdeaImages`, but identical markup and CSS) — Idea's version used to be a distinct older `.idea-image-*` implementation with no Drive-link detection and no drag-reorder; it was unified onto Post's classes/behavior after a report that pasting a Drive folder link in Ideas rendered a broken-image icon instead of the folder chip Post already showed. If you touch one of these two functions, check whether the other needs the same fix.
- **Post/Idea images list doubles as the asset-folder handoff**: there's deliberately no separate "Asset Folder" field anymore (one existed briefly on Posts and was merged back in per feedback that two entry points was confusing). A Google Drive link pasted into the *same* Images input is detected by `isDriveFolderLink()` (matches `drive.google.com`) and rendered as a distinct folder chip (`.image-thumb.folder-link`) instead of a photo thumbnail — full-width (`flex: 1 1 100%` inside the otherwise-square-tile `flex-wrap` grid) with a folder icon and the URL as a CSS-ellipsis-truncated label, since a Drive folder link doesn't encode a human-readable folder name anywhere in it (would need the Drive API to resolve one, which this app deliberately doesn't call). `extractDriveFolderUrl()` pulls just the `https://...drive.google.com/...` substring (trimming trailing punctuation) out of whatever was pasted, so if someone pastes a whole sentence/note that happens to contain a Drive link rather than the bare link, both the label and the actual `href` only ever show/use the real URL, not the surrounding text. Clicking the chip opens the link in a new tab. On Posts it's excluded from the LinkedIn preview (`getPreviewImages()` filters it out) and from the photo-count badge, and is still picked up by `findAssetFolderUrl()` to drive the calendar card's quick-open folder icon. `post.assetFolderUrl` (the old dedicated field) is kept only as a fallback inside `findAssetFolderUrl()` for any posts saved while that field existed — there's no editor for it anymore, don't resurrect one; new Drive links go in Images.
- **Status picker, a custom dropdown**: `#f-status` is a *hidden* `<input>` holding the real value -- the visible UI is `.status-picker` (button + popup `role="listbox"`), because no browser lets you style `<option>` elements as pills. The trigger shows the current value as a real `.status-pill` (identical markup to the calendar's status badges); the popup list shows each option the same way. `updateStatusSelectStyle()` syncs the trigger's pill whenever `fStatus.value` changes (on selection, or when set programmatically on modal open/reset) -- call it any time you set `fStatus.value` directly, since a hidden input doesn't fire native `change` events. (History: first tried skinning the real `<select>` into one big pill -- reverted, unwanted. Then tried just tinting the select's background/border -- reverted again once the user asked for the *options* themselves to be pill-shaped like the calendar, which a native select can't do at all. This custom-dropdown version is the one that stuck.) If you build another "picker where each choice should look like its own status pill," follow this pattern rather than fighting `<select>` styling again.
- **Color-coding icons by category**: when a set of icons represents different *kinds* of the same thing (activity feed action types being the example -- created/updated/deleted/moved/feedback), color them by category using existing tokens rather than leaving them all one muted color, or they blur together at a glance. Pattern: a `ACTIVITY_ICON_CLASS` map alongside the icon map, applied as an extra class on the icon wrapper (`.activity-icon.act-created` etc.), not inline styles. Current mapping: green `--good` = created, brand orange `--brand` = updated, red `--bad` = deleted, blue `--b-case` = moved/stage-changed, indigo `--b-news` = feedback/comment -- the `--b-*` tokens are just spare named colors already in the palette (seed colors for default content buckets), reused here for variety since they aren't live-bound to any specific bucket at render time.

## Tasks (removed from the UI, kept in the data)

The assignable Tasks checklist (on both Post and Idea) was removed from the form for now, per direct feedback that it added clutter. The `tasks` field is still read on load and passed straight back through on save (`currentPostTasks` / `currentIdeaTasks` — plain variables, not editable) specifically so existing saved tasks aren't lost even though there's no editor for them. The `createTaskListInput()` factory is still defined but unused; re-adding the UI later is mostly restoring the HTML block removed from both forms and re-wiring those two variables back to the controller.

## Growth goals (multi-goal)

Goals live in a `goals` collection, **one document per metric, keyed by the metric name as the doc ID** (`goals/followers`, `goals/impressions`, etc.) — this is what makes "one active goal per metric" automatic: saving a goal for a metric is just `setDoc(doc(goalsCol, metric), {...})`, which always overwrites that metric's single doc, so duplicates are structurally impossible without extra validation code. Capped at `MAX_GOALS = 3` total, enforced at save time in the submit handler (checked against `currentGoals.length`, only when the save would *add* a new metric rather than edit an existing one) — there's no cap on the read/render side, so if that constant ever changes, existing over-the-old-cap data (there shouldn't be any) would still render fine.

The Metric dropdown in the modal *is* the goal picker: selecting a metric loads whichever goal already exists for it (or blank fields if none), so there's a single form for both creating and editing rather than separate flows. Clicking a rendered `.goal-card` opens the modal pre-selected to that card's metric. The toolbar "Set Goal" button opens it defaulted to a metric that doesn't have a goal yet, so it naturally starts a new one instead of landing on an existing goal at random.

There's a legacy `settings/goal` single-document goal that predates this (from before multi-goal support) — kept only for a one-time migration (see `goalsMigrationChecked`): the first time the `goals` collection snapshot comes back empty, it reads the legacy doc once and, if it has valid data, writes it into the new collection. Don't remove the legacy read path; it's what stops anyone's already-set goal from silently disappearing when this shipped.

`#goalCardsContainer` is a CSS grid whose column count tracks how many goals actually exist -- `renderGoalCards()` sets its className to `goal-cols-1`/`goal-cols-2`/`goal-cols-3` based on `valid.length` (1 goal = full width, 2 = side by side, 3 = three across), collapsing back to a single column under the same 980px breakpoint `.chart-row` uses. Set via JS class rather than a CSS child-count selector -- simpler, and it's already recomputed on every render anyway.

## Mobile / view-only mode

The app has a real viewport meta tag (`width=device-width, initial-scale=1`) and responsive breakpoints at `980px` and `720px` (see the `@media` blocks near the bottom of the `<style>` block). `720px` is also the line where phones stop being able to create or edit Posts/Ideas — per direct feedback ("On mobile, things will not be editable, only for viewing" -> clarified as "block only Post/Idea editing", not a full lockdown): everything else (Analytics, adding a feedback comment on an Idea, browsing the calendar/kanban) stays fully functional on a phone.

- **`isMobileView()`** (`window.matchMedia("(max-width: 720px)").matches`) is the single source of truth for "are we in the phone breakpoint" on the JS side — always reuse it rather than a second hardcoded width, so the JS gate and the CSS breakpoint can never drift apart.
- **Creation entry points are hidden by CSS alone**, not JS: `#newPostBtn`, `#newIdeaBtn`, `.day-add` are `display: none` inside the `720px` media query. No JS gating needed here since there's nothing to lock, just nothing to show.
- **Opening an existing Post/Idea still works on mobile** — tapping a `.post-card` / `.idea-card` calls the same `openModal()` / `openIdeaModal()` as desktop. What changes is `setPostReadOnly(isMobileView())` / `setIdeaReadOnly(isMobileView())`, called once at the end of each open function (checked at open-time only — resizing/rotating *while a modal is already open* won't flip it live, which is an acceptable edge case, not a bug to fix reactively).
- **Read-only implementation**: text-like fields (`f-title`, `f-copy`, `i-title`, `i-notes`, date/time) get `.readOnly = true` rather than `.disabled`, specifically so the text stays full-contrast and selectable/copyable — this is a *view*, not a disabled form. Fields with no read-only concept (`<select>`, the custom status-picker trigger `<button>`) get `.disabled = true` instead, which is the only option available for those control types. The add-rows and remove-buttons for owners/images/emoji (`.owner-input-row`, `.image-link-row`, `.idea-link-row`, `.owner-chip-remove`, `.image-remove`, `#emojiToggleBtn`, `#ideaEmojiToggleBtn`) are hidden entirely via one shared `.modal.view-only` CSS rule rather than disabled one-by-one in JS, since there's nothing meaningful to show in a disabled-but-visible state for an "add new chip" row. Save/Delete/Send-to-Calendar buttons are hidden (`.hidden = true`), and Cancel's label flips to "Close" so the one remaining action reads correctly.
- **A `.view-only-badge` ("View only" pill)** sits next to the modal title (`#postViewOnlyBadge` / `#ideaViewOnlyBadge`, inside a `.modal-head-title` wrapper alongside the `<h3>`) so the locked state is visually obvious, not just implied by fields being unresponsive.
- **Idea feedback is the deliberate exception**: `i-feedback-input` and `#addFeedbackBtn` are never touched by `setIdeaReadOnly()` and stay fully live on mobile. Adding feedback calls `updateDoc()` directly (it's not routed through the Idea form's own submit handler at all, mobile or not), which is exactly why it can safely stay enabled while the rest of the idea is locked — there's no "save the whole form" step it depends on.
- **Autofocus is skipped in read-only mode**: `openModal`/`openIdeaModal` normally autofocus the title field on open; on mobile this was actively harmful, since focusing a field scrolls the (often taller-than-viewport) modal so the field is in view, which shoved the "View only" badge and close button off the top of the screen before the user ever saw them. Both autofocus calls are now wrapped in `if (!isMobileView())`.
- **Narrow-phone layout polish**: any flex row of several buttons (topbar actions, the Analytics header's Set Goal/Export/Import row) needs `flex-wrap: wrap` or it silently forces the whole page wider than the viewport — a flex parent wrapping its own children doesn't help if a *child* row further down refuses to wrap its own contents. `.day-cell`/`.week-col` got `overflow: hidden` at the `720px` breakpoint for the same reason: the month/week grid's columns are sized with an explicit `minmax(2.4rem, 1fr)`, so a post-card's meta row (avatars, badges, status pill) that doesn't fit will overflow its cell rather than stretch the grid — `overflow: hidden` clips that instead of letting it blow out the page's scroll width. When adding new UI, verify with `document.documentElement.scrollWidth <= document.documentElement.clientWidth` at a real phone width (390px), not just eyeballing a screenshot — content can silently exceed the viewport without an obvious visual cue if it's the last element in a row.

## Schema changes

Any new field on `posts`/`ideas` must be additive and read with a fallback (`post.newField || defaultValue`) — existing documents without it should degrade gracefully, never break. See `normalizedGoal()` for the pattern used when a field's *shape* changes, not just its presence.
